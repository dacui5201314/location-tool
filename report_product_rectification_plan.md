## 二、报告产品价值判断

用户购买或消耗点数生成报告时，真正想要的不是一篇很长的分析文章，而是更快判断：

1. 这个位置值不值得继续看。
2. 最大机会是什么。
3. 最大风险是什么。
4. 周边竞品到底多不多。
5. 接下来现场应该重点核验什么。
6. 报告数据是否有依据。
7. 如果系统失败，是否扣点，是否退款，是否值得再试一次。

所以产品目标不是“输出更多文字”，而是交付一份能降低选址风险的初筛工具：

- 看得懂。
- 信得过。
- 有依据。
- 可行动。
- 不过度承诺。
- 失败时让用户安心。

## 三、理想报告结构

后端 report_json 目前已经围绕以下用户可见结构收口。前端和 HTML 后续只能展示这些字段，不能自行生成业务判断。

### 1. 首屏决策卡

首屏必须优先回答：

- 总体判断：可优先现场核验 / 谨慎考察 / 低优先级候选点。
- 总分。
- 一句话结论。
- 最大优势。
- 最大风险。
- 下一步最重要的现场动作。
- 适合成立的条件。
- 应该停止推进的条件。

表达必须克制，禁止“推荐开店”“值得投资”“强烈建议入驻”等决策替代语言。

### 2. 关键证据摘要

用户必须能快速看到真实半径数据：

- 直接竞品：200m / 500m / 1000m。
- 替代消费：200m / 500m / 1000m。
- 客流锚点：200m / 500m / 1000m。
- 住宅、学校、办公、交通、商场、医院、停车等关键 POI。

所有数据都要明确半径，避免“周边很多”“附近较少”这类无半径模糊表达。

### 3. 行业口径说明

报告需要解释：

- 什么算直接竞品。
- 什么算替代消费。
- 什么算客流锚点。
- 哪些行业存在暗竞品或 POI 漏收录风险。
- 为什么 0 竞品不一定是机会。

这部分是用户建立信任的关键。尤其是小吃快餐、正餐、茶饮、教育托管、教育培训、酒店、美业宠物、社区服务等行业，不能套同一套泛化逻辑。

### 4. 地点基本面 + 生意模型快照

报告要先说明“这个地方本身是什么位置”，再说明“这个业态在这里是否匹配”。

地点基本面示例：

- 学区边缘。
- 社区底商。
- 办公商圈。
- 弱交通住宅区。
- 交通节点。
- 商业综合体外围。

生意模型快照应按 12 个业态族群输出不同判断，不允许把教育托管写成餐饮逻辑，也不允许把 0 POI 竞品简单写成蓝海。

### 5. 现场核验清单

每份报告都应输出可执行的现场核验任务。核验任务应根据业态变化：

- 小吃快餐：午市/晚市客流、出餐速度、外卖取餐、门头可见度、强替代分流。
- 正餐聚餐：停车、排烟、消防、晚市聚餐需求、包间/翻台。
- 茶饮烘焙：年轻客群、步行动线、外卖平台强品牌、堂食停留。
- 便利超市：主出入口动线、入住率、晚间/早间便利需求。
- 药店：医院/社区支撑、医保/处方/合规、周边药店密度。
- 目的零售：商圈层级、楼层动线、品类匹配、低频目的性。
- 教育托管：目标小学、放学动线、家长接送、安全合规、暗竞品。
- 教育培训：家庭消费力、课程需求、满班率、续费率、品牌信任。
- 社区基础服务：暗竞品、资质、社区刚需、复购。
- 美业健身宠物：工作室暗竞品、物业限制、噪音/气味、会员转化。
- 酒店：过夜需求、交通可达、商旅/旅游源、淡旺季。
- 休闲娱乐：夜间交通、扰民、合规、停车和客单。

### 6. 数据边界

每份报告都必须明确：

- 数据来源于地图 POI + 系统规则分析。
- POI 可能存在更新延迟。
- 报告只用于选址初筛。
- 不替代现场调研、租金测算、合同审查和实际投资决策。
- 租金、客流、营收、转让费、证照、消防、物业限制需要现场确认。

## 四、前端与管理后台下一阶段重点

后端报告判断力主线已经完成，下一阶段重点转向“把能力包装得更好用、更像能卖钱的产品”。

### 2026-06-18 首轮体验优化进展

- 小程序首页流程已按“选位置 -> 选业态 -> 填画像 -> 出报告”重排。
- 小程序报告详情已按用户阅读顺序重排：综合评分置顶，优势/风险上下排列，现场核验清单、竞品口径和数据边界重新整理。
- 管理后台 IA 已分组，报告库预览改为结构化渲染 `report_json` 并向小程序端对齐。
- 用户反馈闭环已接入：前端提交、后台处理回复、用户端查看回复。
- 本阶段没有改变报告判断逻辑、YAML、guard、prompt 或 `/api/analyze`。

### P0：小程序前端体验

重点优化范围：

- 首页选址流程更清晰：地址、业态、品牌/特色、面积、生成按钮的层级更顺。
- 生成前预期更明确：消耗点数、失败退点、报告用途边界。
- 业态选择更友好：搜索、分组、示例、避免用户选错大类。
- 报告详情首屏更有判断力：先看结论、优势、风险、下一步动作，再看详细数据。
- 加载态、失败态、空状态更安心。
- 收藏、充值、个人中心等页面统一视觉和状态反馈。

### P0：管理后台信息架构

当前管理后台主要问题不是某个按钮丑，而是模块混杂。下一窗口应优先整理 IA：

- 仪表盘：经营概览、报告生成、失败率、fallback 比例。
- 用户与订单：用户、点数、会员、订单、退款/补偿。
- 报告运营：报告库、失败原因、fact_guard 原因、fallback 报告。
- 业态配置：前台业态、分类、启停、展示顺序。
- AI/分析配置：系统 prompt、行业 prompt、模型配置。该区必须标为高风险配置。
- 地图与数据源：高德 Key 池、调用状态、额度。
- 支付配置：微信支付、虚拟支付、SKU/CDK。
- 系统设置：分享配置、客服二维码、基础参数、审计。

管理后台优化可以改菜单、布局、表格、表单、状态提示；不得借后台优化修改报告生成语义。

### P1：展示一致性和视觉统一

- 小程序与 HTML 继续使用同一份 report_json。
- 报告模块标题、字段含义、免责声明、数据边界保持一致。
- 按业务重要性折叠低频细节，避免手机端信息过载。
- 统一按钮、卡片、表单、标签、空状态、错误态。
- 避免让弱相关 POI 淹没核心判断。

### P2：后续技术与运营能力

这些能力可以作为后续版本，不阻塞当前前端/后台优化：

- 质量投诉样本闭环。
- 后台健康度看板。
- `/api/analyze` 主链路拆分专项。

其中 `/api/analyze` 目前虽然较长，但测试全绿，短期不建议重构。应在前端和后台稳定后，单独开后端链路拆分专项处理。

## 五、后续资料蒸馏流程

如果后续找到高质量书籍、报告或行业资料，必须按固定流程进入系统，不能直接写进 YAML 或 prompt。

标准流程：

1. 资料接入：记录来源、文件、版权边界、适用业态。
2. source card：先建立独立卡片，标注 `candidate_only` 或 `absorbed`。
3. manifest：登记 source_id、source_type、title、confidence、distillation_status。
4. 候选规则：未验证资料只能写入 candidate_rules，不进入业务模型。
5. YAML 吸收：只有高质量、可追溯、可测试的规则才能进入 business_model YAML。
6. source_refs：每条吸收规则必须补 source_refs。
7. service/report_json：如规则影响输出，必须同步 service 或 enrichment。
8. HTML/小程序展示：只消费 report_json，不各自生成判断。
9. 测试：schema、business_model、sample_regression、fact_guard 必须补齐。

当前没有高质量书源时，不强行吸收低质量资料。可以先把资料列为 candidate source card，等待后续验证。

## 六、禁止回退项

以下红线继续有效：

- 不重构 prompt 主体来掩盖业务模型问题。
- 不放宽 `report_fact_guard.py`。
- 不放宽 `poi_name_guard.py`。
- 不让 HTML 或小程序自己生成业务判断。
- 不为了赶进度把未验证资料直接吸收到 YAML。
- 不把 0 竞品简单写成蓝海或强机会。
- 不把学校 POI 对所有业态统一加权。
- 不把公交线路数当成公交站点数。
- 不把教育托管、教育培训、餐饮、生活服务套同一套现场核验清单。
- 不在前端/后台优化窗口修改报告评分、POI 分类、YAML、guard 或 prompt 语义。

## 七、当前建议执行顺序

1. 同步当前后端和小程序构建产物到服务器/微信开发者工具。
2. 在服务器执行 `database.init_db()`，重启后端，确认 `/api/health` 正常。
3. 用真实账号验证：生成报告、报告详情、提交反馈、我的反馈、后台回复、报告库预览、HTML 下载。
4. 检查小程序审核相关文案，确保登录页不出现混淆腾讯官方的元素。
5. 若继续优化，只做前端/后台展示和运营体验，不继续改报告判断逻辑。
6. 前端/后台稳定后，再单独评估 `/api/analyze` 主链路拆分专项。

## 八、最终判断

址得选报告整改的第一阶段已经完成：系统已经从“生成一篇分析”推进到“基于知识框架输出选址初筛判断”。

接下来真正影响用户感知价值的，不是继续堆更多后端规则，而是把现有能力以更清晰的前端流程、更可信的报告详情、更易用的管理后台呈现出来。

报告产品的核心标准保持不变：

- 首屏能判断。
- 数据能解释。
- 现场能执行。
- 失败不焦虑。
- 结论不过度承诺。
- 所有判断都有 report_json 和 source_refs 追溯。
