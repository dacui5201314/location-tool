# 2026-06-22 报告修复方案

## 一、修复背景

本轮问题集中在三条链路：

1. 新生成报告在管理后台详情页一直显示“加载中”，对象存储桶中也没有对应报告文件。
2. 报告周边数据中“购物商场”识别为 0，但实际场景中商场、市场、专业市场都可能对选址判断有价值。
3. “各维度详细分析”存在模板化废话，例如“数据来自地图服务点位采集，已做名称脱水处理”，用户读不到明确判断。

补充需求：

1. 市场类点位不能简单丢弃，应在数据质量或数据解释板块中体现其参考价值。
2. 前端可考虑增加“预估租金”可选填写项，让成本与盈亏测算更精准。

## 二、修复边界

本方案只定义修复方向与验收标准，后续实施时需遵守以下边界：

1. 不随意放宽 `report_fact_guard.py`、`poi_name_guard.py`。
2. 不直接改动总分、维度评分、竞品口径和 POI 分类语义，除非有测试锁定正反例。
3. 小程序、HTML、管理后台继续消费同一份 `report_json`，不得各自生成业务判断。
4. 对报告表达做白话化优化时，优先调整展示与模板文案；涉及 prompt 的修改必须配回归样本。

## 三、问题一：后台详情加载与对象存储

### 现象判断

生成链路设计上是先写数据库 `report_json`，再生成报告 HTML 并上传对象存储。即使云上传失败，也应保留本地兜底文件，并在后台通过 DB JSON 预览。

因此“桶里没有文件”不应该直接导致后台空白。管理后台一直显示“加载中”，更可能是：

1. 详情接口请求失败、超时或返回非 JSON。
2. 后台 `viewReport()` 没有捕获渲染异常。
3. `arRenderReport()` 遇到非预期字段结构后 JS 报错，中断 DOM 更新。
4. 线上新报告记录存在 `report_json` 为空、解析失败、`report_file/report_url` 都缺失的异常状态。
5. 云存储配置误用：后台提供阿里云 OSS 与腾讯云 COS 入口，但后端实际上传链路只实现腾讯云 COS。

### 修复方案

1. 后台详情前端增加分层容错：
   - `viewReport()` 对接口请求增加 `try/catch`。
   - `arRenderReport()` 渲染失败时展示错误卡片，不停留在“加载中”。
   - 错误卡片展示 `report_uuid`、`storage_source`、`parse_error`、`storage_error`。
   - 保留“原始 JSON”折叠面板，方便运营和技术排查。

2. 后台详情接口增强诊断字段：
   - 返回 `report_json_length`。
   - 返回 `has_report_json`、`has_report_file`、`has_report_url`。
   - 当 `report_json` 为空但有本地/COS HTML 时，明确 `storage_source`。
   - 当三者都缺失时，返回可读错误原因。

3. 对象存储配置收紧：
   - 后台如暂不支持阿里云 OSS，应隐藏或禁用“阿里云 OSS”选项。
   - 保存云存储配置时，`storage_provider` 必须为 `tencent_cos` 才允许测试上传。
   - 配置保存后必须跑健康检查上传，失败时提示“云存储不可用，不会写入存储桶”。
   - 报告库列表展示“DB JSON / 本地兜底 / COS / 缺失 / 云上传失败”。

4. 报告保存链路保留现有策略：
   - DB 主记录保存失败时，不返回成功。
   - 云上传失败时，保留本地 HTML，并将 `_storage_error` 写入 `report_json`。
   - 不因为 COS 失败阻断用户查看报告，除非 DB JSON 也失败。

### 验收标准

1. 新报告即使 COS 上传失败，后台也能通过 `report_json` 打开结构化预览。
2. COS 配置错误时，报告库能明确显示“云上传失败，本地兜底”或“仅 DB JSON”。
3. 详情页任何接口/渲染异常都展示错误卡片，不再无限“加载中”。
4. 存储配置测试成功后，桶中能看到 `healthcheck/` 和新报告 `reports/` 文件。

## 四、问题二：购物商场、市场与有价值点位

### 现状问题

当前“购物商场”脱水规则偏窄，只保留购物中心、百货、商场、商城、商业广场、步行街、购物公园、MALL、万达广场、吾悦广场、银泰城、奥特莱斯等。

这会导致一些真实商业体被剔除，例如：

1. 万象城、万象汇。
2. 龙湖天街。
3. 大悦城。
4. 印象城。
5. 宝龙广场。
6. 生活广场。
7. 商业中心、商贸中心。

同时，市场类点位也不应全部视为噪音。农贸市场、批发市场、建材市场、五金市场、家居市场、汽配城、菜市场等，虽然不等同于购物中心，但能体现目的性客流、价格敏感客群、早晚高峰和专业采购客流。

### 分类方案

不建议把市场简单合并进“购物商场”。建议拆成三层：

1. `shopping_mall`：购物中心、百货、商业综合体、商场、商城、步行街、天街、万象城、大悦城、吾悦广场、宝龙广场等。
2. `market_anchor`：农贸市场、菜市场、批发市场、建材市场、五金市场、家居市场、汽配城、商贸城等。
3. `low_freq_retail`：服装、数码、手机、家电、眼镜、珠宝、名创优品、屈臣氏等目的性零售。

展示上可以继续保留用户熟悉的“购物商场”，但数据解释中要补充“市场/专业市场”：

1. “购物商场”代表逛街、休闲、品牌消费和综合客流。
2. “市场/专业市场”代表目的性采购、价格敏感消费、早市/晚市或行业集散客流。
3. “低频目的零售”代表非高频但有消费力的目的性到访。

### 数据质量板块优化

将“数据质量”升级为“数据解释与边界”，展示更有价值的信息：

1. 有效保留点位总数。
2. 名称脱水剔除数量。
3. 严谨度规则剔除数量。
4. 市场/专业市场数量与样例名称。
5. 低频目的零售数量与样例名称。
6. 便利店/超市数量。
7. 停车场、银行、酒店等辅助设施。
8. 医院归并、公交去重等处理说明。

示例表达：

> 500 米内未识别到购物中心类商业体，但识别到 2 个市场/专业市场。它们不等同于商场客流，但说明周边存在目的性采购客流，需现场确认人流时段、消费客群和门店业态是否匹配。

### 采集与识别修复

1. 增加购物商场专项补采：
   - 类型码补采：购物中心、百货、综合商场相关类型。
   - 关键词补采：购物中心、商场、商业广场、万象城、天街、大悦城、吾悦广场、宝龙广场、步行街等。

2. 增加市场/专业市场专项补采：
   - 关键词补采：农贸市场、菜市场、批发市场、建材市场、五金市场、家居市场、汽配城、商贸城。
   - 不参与“购物商场”计数，但进入 `market_anchor` 与数据解释。

3. 扩展正反例测试：
   - 正例：万象城、龙湖天街、大悦城、印象城、宝龙广场、生活广场、商业中心。
   - 市场参考正例：农贸市场、菜市场、建材市场、五金市场、家居市场、汽配城。
   - 反例：普通广场、广场舞场地、停车场、便利店、超市、维修店、彩票店。

### 验收标准

1. 真实商业体不再大量显示为 0。
2. 市场类点位不污染“购物商场”计数，但能在“数据解释与边界”中体现。
3. 小程序、HTML、后台三端展示口径一致。
4. 商场/市场正反例测试全部通过。

## 五、问题三：各维度详细分析白话化

### 现状问题

部分详情文案把数据来源说明塞进维度正文，例如：

> 数据来自地图服务点位采集，已做名称脱水处理。

这类句子对用户判断选址没有直接帮助，应放到数据说明或边界板块，不应占据“人口密集度”等分析正文。

### 修复方案

1. fallback 模板改写：
   - 删除维度正文中的数据来源说明。
   - 将数据来源、更新延迟、名称脱水处理统一放到 `data_boundary` 或“数据解释与边界”。

2. 维度正文统一结构：
   - 先说事实：500 米内住宅、办公、学校、医院等数量。
   - 再说判断：这说明客源基础强、一般还是偏弱。
   - 再说验证：现场应看什么。
   - 最后说降级条件：什么情况下不应继续看。

3. prompt 增加表达约束：
   - 维度分析禁止写“数据来自”“已脱水”“地图服务采集”等数据来源句。
   - 每段必须输出用户能理解的经营含义。
   - 少用系统术语，多用白话判断。
   - 不改变事实校验、评分、POI 口径。

### 示例改写

原文：

> 500 米半径内 17 个住宅小区、0 栋办公建筑、4 所教育机构、0 家医院。数据来自地图服务点位采集，已做名称脱水处理。如现场发现数据中未收录的新建小区或写字楼，人口密度可能高于采集结果。

建议：

> 500 米内有 17 个住宅小区，但办公和医院客流基本缺失，说明这里更像社区生活圈，而不是办公或医疗带动型点位。对餐饮、便利零售这类高频业态，重点看小区入住率、晚间人流和周边居民消费能力；如果小区入住率低或门前动线不经过，该点位的人口支撑要下调。

### 验收标准

1. “各维度详细分析”不再出现“数据来自”“名称脱水”“POI 采集”等模板化数据来源句。
2. 每个维度都能回答“这说明什么、怎么验证、什么情况降级”。
3. 低分维度解释清楚为什么低，不只罗列数字。
4. 小程序、HTML、后台展示一致。

## 六、新增建议：前端租金可选填写

### 是否值得做

值得做。当前前端只收集品牌/特色和门店面积，后端请求也只有 `store_size`。报告中的房租成本和盈亏测算主要依赖模型估算，所以成本维度容易被标记为低置信。

增加租金输入后，报告可以更准确地判断：

1. 月租金压力。
2. 单平租金是否偏高。
3. 租金占预估营收比例。
4. 盈亏平衡单量。
5. “是否值得继续谈”的成本边界。

### 前端设计

在“品牌与面积”步骤下增加可选字段：

1. 字段标题：租金信息（可选）
2. 输入模式：
   - 月租金：元/月。
   - 单平租金：元/㎡/月。
3. 默认不填写，不阻断生成报告。
4. 帮助文案：
   - “填了会让成本和盈亏测算更准，不确定可留空。”

建议字段：

1. `monthly_rent`：整店月租金。
2. `rent_per_sqm`：单平月租金。
3. `rent_input_type`：`monthly` 或 `per_sqm`。

### 后端设计

1. `AnalyzeRequest` 增加可选字段：
   - `monthly_rent: Optional[float] = None`
   - `rent_per_sqm: Optional[float] = None`
   - `rent_input_type: str = ""`

2. 统一换算：
   - 如果用户填 `monthly_rent`，直接使用月租。
   - 如果用户填 `rent_per_sqm` 且有 `store_size`，计算月租为 `rent_per_sqm * store_size`。
   - 如果两者都填，优先使用用户选择的输入模式，并保留另一个作为参考。

3. 写入报告：
   - 写入 `report_json.cost_inputs`。
   - 写入 `report_json.real_data.user_cost_inputs` 或独立 `cost_inputs`。
   - 管理后台详情展示“用户填写租金”。

4. 报告生成：
   - prompt 中明确标注“用户填写租金，不是模型估算”。
   - fallback 成本详情优先使用用户租金。
   - `dimension_score_meta.cost_estimate` 如果已有租金输入，应减少“低置信”提示。

### 验收标准

1. 不填租金时，当前报告生成流程不受影响。
2. 填月租金时，报告成本维度能直接引用月租，并给出租金压力判断。
3. 填单平租金和面积时，报告能换算月租。
4. 小程序、后台、HTML 都能展示租金输入来源。
5. 成本维度置信度提示能区分“用户填写租金”和“模型估算租金”。

## 八、测试清单

1. 后台报告详情：
   - 正常 DB JSON 报告可打开。
   - COS 失败但 DB JSON 存在时可打开。
   - `report_json` 坏数据时展示错误态。
   - `report_json/report_file/report_url` 都缺失时展示缺失态。

2. 对象存储：
   - COS 配置正确时，`healthcheck/` 上传成功。
   - 新报告上传到 `reports/`。
   - COS 配置错误时报告本地兜底，并写入 `_storage_error`。

3. 商场/市场识别：
   - 万象城、龙湖天街、大悦城、宝龙广场进入购物商场。
   - 农贸市场、建材市场、五金市场进入市场/专业市场参考层。
   - 建材市场不进入购物商场。
   - 便利店、超市不进入购物商场。

4. 维度文案：
   - `details` 中不出现“数据来自”“名称脱水”“POI采集”。
   - 每个维度有事实、判断、验证、降级条件。

5. 租金输入：
   - 不填租金可正常生成。
   - 填月租金可生成并展示。
   - 填单平租金可按面积换算。
   - 成本维度置信度随租金输入变化。

5. 检查后台报告库详情是否能打开。

7. 检查“购物商场”“市场/专业市场”“数据解释与边界”展示。
8. 检查各维度详细分析是否为白话判断。
9. 检查成本/租金相关字段是否正确进入报告。
