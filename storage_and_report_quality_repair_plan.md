# 云存储与报告质量修复方案

## 1. 背景

本次修复针对两个 launch 前必须闭环的问题：

1. 用户生成的文件仍未稳定保存到腾讯云对象存储。
2. 用户生成报告出现事实口径、表达归类、矛盾解释、评分依据不一致的问题。

典型问题包括：

- 数据表显示 `500m` 内公交站为 `2` 个，报告却写成 `500m` 内 `5` 个公交站点；`5` 实际来自 `1000m` 数据。
- `79` 家餐饮、`14` 家酒店说明外圈商业生态活跃，但 `500m` 内仅 `1` 个小区、`0` 栋写字楼，报告没有解释客源来自哪里。
- “关键优势”中混入大量风险和限制条件，导致优势看起来像自我否定。
- 人口密集度、成本压力等评分缺少足够依据，尤其“未做营收测算”时仍给出成本压力 `50` 分。

## 2. 修复原则

- 不硬改 prompt 作为补丁。
- 不针对单个品牌、单个城市、单个样本写特例。
- 不大幅重调全行业评分。
- 不覆盖已蒸馏的业态知识和 12 族分支，而是在事实口径、守卫、表达归位、缺失值处理上加护栏。
- 先修确定性链路，再考虑提示词和深层模型逻辑。

## 3. 总体顺序

报告质量链路按以下顺序推进：

1. 事实层
2. 守卫层
3. 表达层
4. 矛盾解释层
5. 评分缺失值层
6. 回归测试

云存储链路可并行修复，因为它与报告生成语义不冲突。

优先级判断：

- 如果只能串行推进，先做报告质量的事实层和守卫层，再做云存储。
- 如果可以并行推进，云存储作为独立工作流同步修复。
- 评分缺失值最后上线，且必须先做新旧评分分布对比。

## 4. 云存储修复方案

### 4.1 根因判断

当前报告保存流程存在几个风险点：

- 云存储配置默认仍可能是本地模式或非腾讯云 provider。
- COS 配置缺失时容易静默退回本地。
- 报告保存只根据返回值是否以 `http` 开头判断写入 `report_url` 还是 `report_file`。
- 用户生成文件不只有报告 HTML，反馈截图、头像、二维码、分享图等也需要统一进入对象存储。

### 4.2 修复动作

1. 建立统一对象存储结果对象：
   - `ok`
   - `url`
   - `key`
   - `provider`
   - `error`
   - `local_path`

2. 改造报告保存：
   - 本地文件仍可作为兜底。
   - 云模式下上传失败必须记录错误，不能静默当作成功。
   - 云上传成功后写入 `report_url`。
   - 本地兜底路径写入 `report_file`，方便后续补传。

3. 后台系统设置增加 COS 配置验证：
   - 保存配置时校验必填项。
   - 提供“测试上传”接口，上传小文件到 `healthcheck/<timestamp>.txt`。
   - 测试文件不使用固定覆盖名，避免刚写入后读取或列表查询产生误判。
   - 查询最近一次测试文件，验证可写、可读、可列举。
   - 返回最近一次验证状态和错误信息。

4. 统一用户生成文件上传入口：
   - 报告 HTML
   - 反馈截图
   - 用户头像
   - 二维码图片
   - 分享图

5. 增加历史补传脚本：
   - 扫描已有本地 `report_file` 但无 `report_url` 的记录。
   - 上传到 COS。
   - 成功后回填 URL。
   - 失败记录原因，不删除本地文件。

### 4.3 验收标准

- 云模式且 COS 配置正确时，新生成报告必须写入 `report_url`。
- 云模式上传失败时，后台能看到明确错误。
- 本地模式仍可正常使用。
- 反馈截图、头像、二维码、分享图不再各自散落为独立本地逻辑。
- 可通过补传脚本迁移历史本地报告。

## 5. 报告质量修复方案

## 5.1 事实层

目标：统一报告中所有用户可见数字的事实来源。

修复动作：

1. 建立 canonical evidence：
   - 保留 `200m / 500m / 1000m` 三个半径的原始统计。
   - 保留去重后的展示统计。
   - 明确每个展示数字来自哪个半径、哪个字段。

2. 公交去重作为第一个实例：
   - 有距离数据时，必须按半径过滤后再去重。
   - Phase 0 先核实 `poi_lists` 和 `traffic_anchor_list` 中公交站条目是否稳定带有 `distance` 字段。
   - 如果采集结果缺少 `distance`，先在采集层补齐距离字段；事实层只能做保守上限约束，不能假装已经完成半径过滤。
   - 没有距离数据时，`500m` 去重结果不能大于 `stats_500m.bus`。
   - 禁止把 `1000m` 的公交站数量写成 `500m` 优势。

3. 核实并统一地点基本面视图：
   - 先确认 `location_profile` 和 `location_fundamentals` 是否同源。
   - 如果二者只是同一底层数据的不同展示视图，不做字段合并，只统一两个视图的数据引用路径。
   - 如果确实存在独立计算逻辑，则让二者都从 canonical evidence 取数，避免旧字段与新字段矛盾。
   - 迁移前必须枚举全部调用方，尤其是 fallback 报告、执行摘要、admin 展示和小程序展示。
   - 不直接删除旧函数；先保证所有调用方都切到同一事实口径，再考虑是否保留兼容包装。
   - admin 和小程序展示优先使用统一后的事实字段。

验收标准：

- 同一个半径、同一个指标，在报告正文、地点基本面、优势、数据表、HTML 中数值一致。
- 附件样例中不再出现“500m 内 5 个公交站”的错误。

## 5.2 守卫层

目标：事实口径统一后，再用 guard 拦截错误引用和半径错配。

修复动作：

1. 扩大 guard 扫描范围：
   - `details`
   - `advantages`
   - `disadvantages`
   - `executive_summary`
   - `action_plan`
   - `summary`
   - `location_profile`
   - `location_fundamentals`
   - `business_model_snapshot`
   - `decision_snapshot`
   - `field_checklist`
   - `evidence_summary`

2. 增加半径错配硬校验：
   - 句子写 `500m`，就只能引用 `500m` 数据。
   - 如果报告值等于 `1000m` 数据但不等于 `500m` 数据，直接判定失败。

3. 小数字严格校验：
   - 对 `0-3` 这类低计数指标，不再允许宽松容差。
   - 精确表达必须精确匹配。

4. 定义 guard 失败后的行为：
   - 优先进行确定性修正，例如删除或改写明显半径错配的句子。
   - 如果是模型生成内容导致的事实错配，只允许最多一次重新生成或重组。
   - 第二次仍失败时，不向用户交付该报告，返回“报告数据校验未通过，请重新生成或联系客服”的用户提示。
   - 后台记录失败字段、原句、期望值、实际值、报告 ID 和重试次数。
   - 禁止无限重试，避免报告生成死循环。

验收标准：

- `500m=2 / 1000m=5` 时，报告写 `500m 内 5 个` 必须被 guard 拦截。
- guard 只负责事实和半径，不负责表达优劣判断。

## 5.3 表达层

目标：把优势、风险、待核验条件归位，避免优势自我否定。

修复动作：

1. 优势字段只保留明确正向条件：
   - 近场同品类少。
   - 交通可达性强。
   - 周边存在明确客流锚点。

2. 风险字段承接负面和限制条件：
   - 近场住宅弱。
   - 办公支撑不足。
   - 外圈竞争密集。
   - 需核验是否由低客流导致近场空档。

3. 增加表达归位规则：
   - 优势中出现 `但`、`不过`、`同时需` 等转折词时，按转折词拆分。
   - 转折前的正向事实保留为优势。
   - 转折后的限制、风险、待核验内容移入风险或分析摘要。
   - 如果整句话没有可独立成立的正向事实，才整体移出优势。

4. 保留 12 族业态分支：
   - 不把现有分支推倒重写。
   - 仅抽出跨业态通用的表达归位规则。

验收标准：

- “200m 同品类供给较少”可作为优势。
- “1000m 同类门店多、需核验低客流”必须进入风险或分析摘要。
- `decision_snapshot.top_strength` 不再引用带强烈风险语气的句子。

## 5.4 矛盾解释层

目标：当数据生态出现表面冲突时，报告必须解释，不允许直接跳结论。

触发条件示例：

- `1000m` 餐饮、酒店、学校、交通节点活跃，但 `500m` 住宅和办公弱。
- 外圈商业生态强，但近场承接人口弱。
- 竞争供给丰富，但基础客源不足。
- 交通点位不少，但缺少可证明的停留和转化场景。

修复动作：

1. 增加 `demand_source_explanation` 或同等字段。
2. 按业态解释可能客源：
   - 餐饮：学校时段流、道路过路流、外卖半径、酒店短停客。
   - 零售：通勤流、社区补给、目的性消费、临街可见性。
   - 服务：办公/社区稳定需求、复购频次、到店便利性。
   - 休闲娱乐：目的性到访、商圈停留、夜间流量。
3. 当解释缺证据时，明确写为“待现场验证”，不能当作已成立优势。

验收标准：

- 外圈生态热、近场客源弱时，报告必须回答“客从哪来”。
- 解释必须跨业态适用，不能只为餐饮样例服务。

## 5.5 评分缺失值层

目标：处理缺数据时的评分依据和置信度，避免无依据中位分影响总分。

修复动作：

1. 给每个维度增加评分元数据：
   - `score_basis`
   - `score_confidence`
   - `missing_required_inputs`
   - `is_score_applicable`

2. 成本压力修复：
   - 若缺少租金、面积、人工、营收测算，不应固定给 `50` 分。
   - 可标记为“待核验”或低置信中性分。
   - 低置信维度是否参与总分需要显式规则。

3. 人口密集度修复：
   - `500m` 内小区少、办公为 `0` 时，应限制分数上限。
   - 不按单一 `小区数量 + 固定底分` 粗算。
   - 不同业态可保留不同权重，但弱客源不能被外圈餐饮热度直接抵消。

4. 总分兼容：
   - 第一阶段不大幅重调全行业评分。
   - 先处理明显无依据或低置信维度。
   - 保持原有字段兼容，新增解释字段，避免前端断裂。
   - `report_json` 必须向后兼容：新增字段只追加，不删除旧字段，不改变已有字段类型。
   - `dimension_scores[].score` 等既有数值字段保持数值类型；评分依据、置信度、缺失输入放到新增 meta 字段中。
   - 如需更丰富结构，新增 `dimension_score_meta` 或同等并行字段，不把既有 `score` 从 int 改成 object。

5. 新旧评分分布对比：
   - 在修改人口密集度上限、低置信维度是否参与总分前，先拿已有样本跑新旧评分对比。
   - 对比整体均值、中位数、分位数、推荐等级迁移比例。
   - 重点观察原 `30-45` 分低优先级样本是否大面积跌入更低档。
   - 只有分布偏移可解释、可接受时，才上线影响总分的评分规则。

验收标准：

- 报告写“未做营收测算”时，成本压力不会表现为确定的 `50` 分。
- 弱住宅、弱办公场景下，人口密集度不会偏宽松。
- 总分能说明哪些维度是低置信或待核验。
- 评分改动上线前必须有新旧评分分布对比结果。

## 5.6 存量报告处理

目标：覆盖已经生成的错误报告和已经落在本地的文件。

修复动作：

1. 增加报告版本和校验状态：
   - `report_schema_version`
   - `fact_guard_status`
   - `storage_sync_status`
   - `last_revalidated_at`

2. 存量事实问题处理：
   - 批量扫描历史报告中的半径错配高风险句子。
   - 对可确定修复的报告重新生成 HTML 或重新渲染展示结果。
   - 对无法安全修正的报告标记为“需重新生成”。
   - 用户再次打开旧报告时，优先展示已修正版；无法修正时提示报告已过期并引导重新生成。

3. 存量云存储处理：
   - 对本地 `report_file` 但无 `report_url` 的记录执行补传。
   - 上传成功后回填云端 URL。
   - 上传失败保留本地路径和错误原因，供后台追踪。

验收标准：

- 已生成但存在公交半径错误的报告不会继续无提示展示。
- 已生成但只保存在本地的报告可批量补传到 COS。
- 存量处理不会静默覆盖原始记录，必须保留修正时间和状态。

## 6. 测试计划

新增测试：

1. 公交半径错配测试：
   - `200m=1 / 500m=2 / 1000m=5`
   - POI 列表包含不同距离公交站。
   - 断言报告不会写 `500m 内 5 个公交站`。

2. guard 错配测试：
   - 句子写 `500m`，数值取 `1000m`。
   - 必须失败。

3. 优势归位测试：
   - 优势句包含 `但`、`需核验`、`低客流`。
   - 必须拆入风险或待核验。

4. 矛盾解释测试：
   - `1000m` 商业生态活跃，`500m` 住宅/办公弱。
   - 必须生成客源解释或待核验说明。

5. 成本压力缺失值测试：
   - 无租金、无营收、无面积。
   - 不允许固定确定性 `50` 分。

6. 评分分布对比测试：
   - 使用已有样本集生成新旧评分对比表。
   - 输出总分均值、中位数、分位数、推荐等级迁移比例。
   - 人口密集度和低置信维度规则上线前必须通过该对比。

7. 存量报告处理测试：
   - 旧报告可被重新校验。
   - 可修正报告生成新版展示。
   - 不可修正报告标记为需重新生成。

8. 云存储测试：
   - 本地模式。
   - 腾讯云模式成功。
   - 腾讯云配置缺失。
   - 腾讯云上传失败但本地兜底。
   - 历史报告补传。

测试位置和执行方式：

- 后端测试优先沿用 `backend/tests/check_*.py` 的轻量脚本风格。
- 新增测试建议放在 `backend/tests/` 下，按功能命名，例如 `check_report_fact_pipeline.py`、`check_score_distribution_compare.py`、`check_storage_sync.py`。
- 当前仓库根目录未看到正式 CI 配置，先作为本地 release gate 执行；后续若补 CI，再把这些脚本加入 CI。
- `npm.cmd run build:mp-weixin` 属于小程序构建验证，不替代后端回归测试。

回归测试：

- `backend/tests/check_location_profile_rules.py`
- `backend/tests/check_business_model_rules.py`
- `backend/tests/check_p1_business_model_quality.py`
- `backend/tests/check_sample_regression.py`
- `backend/tests/check_report_fact_guard.py`
- `backend/tests/check_report_enrichment_service.py`
- 小程序展示字段变更后运行 `npm.cmd run build:mp-weixin`

## 7. 实施阶段

本阶段安排已将事实层和守卫层合并在 Phase 1 中推进。云存储作为独立闭环可并行执行；若资源只能串行，优先处理报告事实层与守卫层，再处理云存储。

### Phase 0: 现状核实与样本固定

- 固定附件样例为回归样本。
- 核实 `location_profile` 和 `location_fundamentals` 的来源关系。
- 核实 `poi_lists` 和 `traffic_anchor_list` 中公交站条目是否带 `distance` 字段；如缺失，先补采集层数据契约。
- 盘点当前 report、feedback、avatar、qrcode、share image 的保存入口。
- 收集已有报告样本，用于后续新旧评分分布对比。

### Phase 1: 报告事实层与守卫层

- 增加 canonical evidence。
- 修复公交按半径去重。
- 统一 `location_profile` 和 `location_fundamentals` 的数据引用路径。
- 修复 admin/小程序展示优先字段。
- 扩大 guard 扫描范围。
- 增加半径错配硬校验。
- 增加小数字严格校验。
- 定义并实现 guard 失败后的重试、阻断、记录行为。
- 补附件样例回归。

### Phase 2: 云存储闭环

- 抽统一对象存储接口。
- 后台增加配置验证和测试上传。
- 新增历史补传脚本。
- 补 report、feedback、avatar、qrcode、share image 测试。

### Phase 3: 表达层和矛盾解释

- 增加优势/风险/待核验归位规则。
- 转折句拆分时保留转折前的真实优势。
- 增加外圈热、近场弱的矛盾解释。
- 保留 12 族分支，仅抽通用表达规则。

### Phase 4: 存量报告处理

- 批量校验历史报告。
- 对可修正报告生成新版展示。
- 对不可安全修正报告标记为需重新生成。
- 执行本地报告到 COS 的补传。

### Phase 5: 评分影响评估

- 建立新旧评分对比脚本。
- 评估总分分布和推荐等级迁移。
- 确认人口密集度和低置信维度规则的下游影响。

### Phase 6: 评分缺失值上线

- 增加评分依据、置信度、缺失输入字段。
- 修复成本压力无依据固定 50 分。
- 修复弱客源场景下人口密集度偏宽松。
- 只在评分影响评估通过后调整总分计算。

## 8. 风险控制

- 事实层风险低：主要纠正半径和展示口径。
- 守卫层风险低：放在事实层之后做，可减少误判。
- 表达层风险低：只做归位，不删除风险判断。
- 矛盾解释层风险低：增加解释，不改变原始数据。
- 成本压力不再固定 `50` 分风险低：属于明显无依据修复。
- 人口密集度限制分数上限风险中高：会直接影响总分和推荐等级。
- 低置信维度是否参与总分风险中高：涉及总分计算逻辑。
- 评分缺失值层必须先做新旧评分分布对比，再决定上线策略。
- 云存储风险低到中：需注意历史本地文件和线上配置兼容。

## 9. 不做事项

- 不用 prompt 修事实错误。
- 不为附件样例写品牌或地址特例。
- 不直接推翻现有蒸馏知识、行业配置和 12 族分支。
- 不一次性重写 scoring 公式。
- 不让云上传失败静默表现为成功。

## 10. 最终验收

本次修复完成后，应满足：

- 新生成用户文件稳定进入腾讯云对象存储，失败可见、可追踪、可补传。
- 报告中同一指标在正文、优势、地点基本面、数据表、HTML/小程序展示中口径一致。
- 公交站等半径数据不会再出现 `500m` 引用 `1000m` 数值的问题。
- 优势不再混入明显风险和限制条件。
- 外圈生态活跃、近场客源弱等矛盾会被解释。
- 缺少成本或营收测算时，不再给出看似确定的成本压力中位分。
- 所有修复保持全业态适用，不绑定单一案例。
